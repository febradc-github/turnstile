# Turnstile Core Rules

<important>
- Never skip a gate: idea -> design approved -> spec approved -> ready -> todo -> in_progress -> review -> done. Each arrow requires the named approval or check to have actually happened, not been assumed. Per-project settings live in turnstile/config.yml -- read it with `node ${CLAUDE_PLUGIN_ROOT}/scripts/config.js` (one JSON line, defaults when absent) and surface its warnings to the user once per session, never silently. In profile: solo, refine collapses design + spec for leaf items into one approved plan (turnstile/plans/PL-<n>.md, criteria included) and the item lands ready off that single approval; epics use the full breakdown pipeline in both profiles, and the review gate is identical in both. One sanctioned shortcut exists: /turnstile:quick (trivial work and diagnosed bugs, at most quick_max_points points -- config default 3, criteria inline in the item note, one approval) -- it lands at todo or ready directly. Nothing else skips gates, and quick refuses work that deserves the full pipeline.
- The current sprint always lives at turnstile/sprint.yml; completed sprints are archived immutably under turnstile/sprints/. The board YAML holds tracking fields only (id, title, type, parent, status, points, assignee, carryovers, parked_at, notes) -- descriptions and acceptance criteria live in the vault notes and specs, never duplicated into YAML.
- Cancelled work is dropped, never deleted: /turnstile:drop sets status: dropped with a recorded reason. Hand-deleting board entries erases history.
- The board is a two-level hierarchy at most: epic -> story -> task, tracked with optional type and parent fields (absent type means story). Epics and containers (items another item names as parent) never enter a sprint and are never spec'd -- /turnstile:breakdown splits them, and only leaf items flow through the gates.
- Never let the implementer self-certify "done." Only /turnstile:review, via the turnstile-reviewer agent, can move an item to done. One derived exception: when the last child of an epic/story passes review, /turnstile:review flips that parent to done as a rollup of its reviewer-certified children.
- Only one item on the active board may be status: in_progress at a time. /turnstile:work refuses to start a second ticket while another is already in_progress -- finish it (move it to review or done) first, or stash it with /turnstile:park. This keeps /turnstile:review's diff scoped to exactly one ticket's changes.
- /turnstile:park is the sanctioned interrupt valve: it sets the in_progress ticket to status: parked with a parked_at timestamp and a resume note (current state, next step, blockers) in the item note, so urgent unrelated work can start cleanly. Parked items do not count against the one-in_progress rule; /turnstile:pickup offers to un-park the most recently parked one. Park is a stash, not a cancellation -- dropping is /turnstile:drop.
- Interrupts respect the one-in_progress rule: a bug related to the in_progress item is fixed within that item (same diff, same review); an unrelated bug becomes a quick task that runs right after the in_progress item finishes (or right away, if the current ticket is parked first) -- never two half-done things at once.
- Every skill searches the vault's notes (turnstile/brain/, decisions/, architecture/, item notes -- the search_notes MCP tool indexes them all) before starting new work, and surfaces what it finds -- including conflicting notes -- before proceeding.
- Knowledge capture follows the capture config setting. capture: gates (the default): brain-curator is dispatched deterministically at exactly four transitions, each with bounded input, and nowhere else automatically -- (1) review passes: the diff, the ticket, and its criteria; the curator writes decision notes and creates or updates turnstile/code/ notes for the touched files; (2) a design or plan is approved: the approved artifact, to link it and extract any ADR-worthy choice; (3) a ticket is dropped: the recorded reason, captured as a dead-end note; (4) systematic-debugger concludes: the confirmed root cause only, never mid-investigation guesses. capture: opportunistic restores the old behavior: whenever any turnstile skill notices something worth remembering (an architectural decision, a gotcha, a recurring blocker), dispatch brain-curator with a short description -- refine, breakdown, sprint-plan, quick, drop, work, systematic-debugger, and code-reviewer all do this (board and pickup are read-only and never dispatch it; spec is a mechanical translation step with nothing new to surface), and skills that changed code (work, systematic-debugger) include the touched source files so turnstile/code/ notes stay current per pass. In both modes /turnstile:remember files a user-dictated note on demand.
- /turnstile:work follows TDD: write the failing test first, then the minimal code to pass it. Defer to the superpowers:test-driven-development skill if it is installed.
- Commits made by /turnstile:review never include an Anthropic or Claude co-author tag, and never use --no-verify.
- Never read env files. Not .env, not .env.local, not prod.env, not .envrc -- no tool, no shell command, no exceptions, in any skill or agent. They hold secrets. The guard hook blocks access mechanically; never work around it. Need a config value? Ask the user.
- Malformed YAML in any turnstile/*.yml file: surface the parse error and ask the user to fix it by hand. Never guess or auto-repair.
- If a requirement, acceptance criterion, or user request is ambiguous, ask before proceeding. Do not guess.
- Keep chat replies lean: lead with the answer or verdict, include only what changes the user's next action, and never quote file contents, board tables, or these rules back unless asked. After rendering a table (board, pickup), do not recap it in prose.
</important>

## Core values

1. **Token economy.** Keep skill and prompt text concise. Do not re-read files already in the conversation. Dispatch an agent only when independence or isolation is structurally required -- Turnstile has exactly five agents for this reason: turnstile-reviewer (independent judgment), turnstile-coder (context-isolated implementation), brain-curator (cheap isolated note writing), pitch-agent (anchoring-free perspectives), loop-runner (autonomous iteration).
2. **Simplicity first.** Smallest viable file and field structure. No speculative abstraction, no unrequested features.
3. **Anti-hallucination.** Verify any technical or factual claim not directly checkable in the current codebase (web search, cite the source) before asserting it. Below roughly 98% confidence, say so plainly instead of asserting.
4. **No AI sloppiness.** Every markdown file Turnstile produces -- skills, design docs, specs, brain notes -- uses short declarative sentences, active voice, no filler, no hedge-padding, no emoji.
